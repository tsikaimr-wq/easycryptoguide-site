# EasyCrypto 项目全量说明与交易逻辑总览

## 1. 项目定位

这是一个以静态前端为主、以 Firebase 作为账号与数据后端的交易所风格网站项目。  
项目同时包含：

- 官网与营销落地页
- 桌面端登录、注册、Dashboard、资金页、行情页
- 移动端主应用（`mobile.html` 单文件多页面）
- 桌面端永续交易页与交割合约页
- 后台管理台（KYC、充值、提现、合约、客服、用户配置）

从代码实现上看，这不是“撮合引擎 + 清算引擎”型的真正交易所后端，而是：

- 前端负责展示、下单、结算逻辑计算
- Firebase Auth 负责登录注册
- Firestore 负责保存用户资料、订单、充值提现、客服消息、后台控制规则
- 后台管理员通过修改 Firestore 文档来影响用户交易结果和资金状态

这点非常重要，尤其是“永续合约”和“交割合约”的实现方式，都明显是**前端驱动 + Firestore 持久化 + 后台规则干预**，不是服务器侧撮合系统。

## 2. 技术结构

### 2.1 前端技术

- 主要是原生 HTML / CSS / JavaScript
- 大量页面直接内嵌脚本
- 少量公共脚本放在 `js/`
- 没有前端框架构建链，属于静态站点模式

### 2.2 后端依赖

- Firebase Auth：注册、登录、找回密码、短信验证
- Firestore：用户资料、KYC、交易、充值、提现、客服消息、充值地址等

### 2.3 行情与外部数据

- TwelveData：报价、价格刷新、部分桌面/首页/移动端行情
- CoinGecko：价格补充、Rates、Tracker、首页市场预览等
- 二维码生成：`api.qrserver.com`

### 2.4 运行方式

- 本地通过 `server.bat` 启动静态 HTTP 服务
- 发布方式偏向 GitHub Pages / 静态托管
- `firebase-config.js` 直接在前端初始化 Firebase

## 6. 账号、会话与 KYC 门禁

### 6.1 统一门禁入口

`firebase-config.js` 提供三个关键方法：

- `checkSession()`
- `isKycCompleted()`
- `enforceKycAccess()`

### 6.2 门禁逻辑

典型流程是：

1. 先检查 Firebase Auth 是否有登录态
2. 再查 `users/{uid}` 的 `kyc_status`
3. 若未通过 KYC，则跳转 `mobile.html`
4. 若通过，则允许进入桌面 Dashboard / Fund / Trade / Contract / Delivery

### 6.3 实际含义

KYC 是整个项目的业务门闸。  
桌面端很多页面虽然存在，但用户真正进入核心流程前，基本都会被 `enforceKycAccess()` 拦一次。

## 7. 永续合约实现逻辑（重点）

## 7.1 结论先说

本项目里的“永续合约”并不是传统交易所那种服务端撮合、风险引擎、标记价格和清算系统。  
当前实现本质是：

- 前端创建一条 `Perpetual` 类型的 `trades` 文档
- 用户持仓期间，前端用实时价格或后台目标价来展示浮盈浮亏
- 平仓时，由前端读取用户规则、交易覆盖规则、市场价，计算最终 `close_price / pnl / roe`
- 最后把结果写回 Firestore，并直接修改用户余额

也就是说，这套系统更接近“前端结算型永续模拟交易 + 后台可控结果”。

## 7.2 开仓流程

### 移动端

核心函数在 `mobile.html`：

- `handleContractTrade()`
- `fire_submitTrade()`

流程如下：

1. 校验 KYC
2. 读取当前币种、方向、订单类型、价格、手数、TP/SL
3. 计算：
   - `lots`
   - `contract_unit`
   - `notional = lots * contract_unit`
   - `margin`
4. 写入 `trades`：
   - `type = 'Perpetual'`
   - `status = 'Open'`
   - `customClosePrice = null`
   - `admin_outcome = null`
5. 弹出“订单已开启”确认框

### 桌面端

核心函数在 `js/desktop-perpetual-contract.js`：

- `submitDesktopPerpOrder()`

整体逻辑与移动端一致，只是桌面 UI 更完整，包含：

- 开仓表单
- 持仓列表
- 历史列表
- 详情弹窗

## 7.3 持仓展示逻辑

永续持仓展示分两层：

### 第一层：实时行情价格

来源包括：

- 当前页面实时行情
- TwelveData
- CoinGecko 补充

### 第二层：目标结算价展示

代码里专门做了一层“目标结算展示”，核心函数包括：

- `getTradeTargetSettlementMetrics()`
- `computePerpetualClosePriceFromYield()`
- `getPerpetualDisplayPrice()`
- 后台对应的 `buildAdminPerpetualSettlementMetrics()`

它的设计目的不是纯看市场，而是允许系统提前根据后台规则推导一个目标结算结果，并将 UI 朝这个方向平滑展示。

## 7.4 平仓逻辑优先级

移动端 `fire_closePosition()` / 桌面端 `settleDesktopTrade()` 的核心优先级是：

1. **交易级固定平仓价**
   - `trade.customClosePrice`
2. **交易级目标收益率**
   - `trade.customYieldPct` / `customRoePct`
3. **用户级币种永续控制**
   - `users.perpetual_custom_prices[coin].closePrice`
4. **用户级币种目标收益率**
   - `users.perpetual_custom_prices[coin].yieldRate`
5. **遗留全局价格**
   - `users.perpetual_custom_price`
6. **再退回市场价**
   - 当前行情价

特殊点：

- 代码里明确写了：**永续合约优先走价格控制逻辑，不优先走简单 win/loss 强制逻辑**
- 这和交割合约不同

## 7.5 盈亏计算

平仓时统一会算：

- `rawPnl`
- `fee_total`
- `fee_close`
- `final_pnl`
- `realized_roe`

计算方式本质是：

- Long：
  - `(close - entry) / entry * notional`
- Short：
  - `(entry - close) / entry * notional`
- 再减去手续费得到 `final_pnl`
- `final_pnl / margin` 得到 `realized_roe`

## 7.6 余额更新

永续平仓成功后，前端会直接：

- `updateDoc(trades/{id}, { ...结算字段 })`
- `updateDoc(users/{uid}, { balance: increment(pnl) })`

这意味着：

- 余额不是后台异步清算
- 而是前端在平仓时直接写用户余额

## 7.7 后台如何控制永续

后台主要有两条控制线：

### 用户级控制

在用户编辑弹窗中可配置：

- `perpetual_custom_price`
- `perpetual_custom_prices[coin].closePrice`
- `perpetual_custom_prices[coin].yieldRate`

这会影响该用户未来打开或关闭的永续仓位的目标结算价。

### 交易级控制

在合约 Operate 弹窗中可针对某一笔仓位设置：

- `customClosePrice`

如果该仓位已经 Closed，后台再次改价格时，还会：

- 重新算 settlement metrics
- 比较新旧 `final_pnl`
- 把差额通过 `balanceDelta` 回写到 `users.balance`

这说明后台不仅能影响未平仓结果，也能**追溯改写已平仓永续结果**。

## 8. 交割合约实现逻辑（重点）

## 8.1 本质判断

虽然页面名叫“Delivery Contract”，但从代码实现来看，这一模块更像：

- 固定时长
- 固定最低金额
- 到期判断涨跌
- 赢则按固定 payoutRate 结算
- 输则亏掉本金

因此它更接近**定时涨跌合约 / 二元结果合约**，而不是传统交割期货。

## 8.2 交易规则表

在 `mobile.html` 中写死了 `deliveryTradeRules`：

- `30s`
- `120s`
- `300s`
- `1h`
- `12h`
- `48h`
- `7d`
- `15d`

每档包含：

- `minAmount`
- `payoutRate`
- `seconds`

示例：

- 30s / 120s：最低 100 USDT，赢面收益率 30%
- 12h / 48h：最低 10000 / 50000 USDT，收益率 50%
- 7d / 15d：收益率 100%

## 8.3 开仓流程

核心函数：

- `openDelivConfirm()`
- `confirmDelivOrder()`
- `startDelivCountdown()`

流程如下：

1. 用户选择方向、时长、金额
2. 检查该时长对应的最低金额
3. 检查可用余额
4. 写入 `trades`：
   - `type = 'Delivery'`
   - `delivery_time`
   - `payout_rate`
   - `status = 'Open'`
   - `admin_outcome = inheritedRule`
5. 前端启动倒计时

## 8.4 结算逻辑

到期时执行 `showDelivResult()`：

优先级如下：

1. 如果 `admin_outcome === 'win'`
   - 强制赢
   - 视觉上调整结算价，让它符合赢的方向
2. 如果 `admin_outcome === 'loss'`
   - 强制输
3. 否则按市场自然判断：
   - Long：`settlePrice >= purchasePrice` 算赢
   - Short：`settlePrice <= purchasePrice` 算赢

收益计算：

- 赢：`grossPnl = amount * payoutRate`
- 输：`grossPnl = -amount`
- 最终：`profit = grossPnl - feeTotal`

然后写回：

- `status = Win / Lose`
- `settlement_price`
- `fee_total`
- `raw_pnl`
- `final_pnl`
- `pnl_amount`

最后直接更新 `users.balance += profit`

## 8.5 后台如何控制交割合约

交割合约更偏“结果控制型”，后台主要通过：

- 用户级 `users.delivery_rule`
- 交易级 `trades.admin_outcome`

来强制当前或后续交割合约：

- `normal`
- `win`
- `loss`

和永续不同，交割合约并不以“后台目标平仓价”为主，而是以“强制结果”为主。

## 9. 后台控制体系（前后台怎么联动）

## 9.1 用户控制

后台在 `admin.html` 中可编辑用户：

- 基础信息
- 余额
- 状态
- KYC 状态
- 永续规则
- 交割规则

保存后直接写 `users/{uid}`。

## 9.2 合约控制

后台监听 `trades` 集合后，会在 Contracts 区展示：

- 类型：Perpetual / Delivery
- 状态：Open / Pending / Win / Lose / Closed 等
- 当前显示的结算价与盈亏
- Operate 按钮

后台可进行：

- `approveContract()`
- `rejectContract()`
- `openOperateModal()`
- `saveContractOutcome()`
- `restoreContractMarketPrice()`

对于永续：

- 可设定 `customClosePrice`
- 可恢复市场价

对于交割：

- 可设定 `admin_outcome = win/loss`

## 9.3 KYC 审核

后台监听 `kyc_submissions`：

- `approveKYC(uid)`
  - `kyc_submissions.status = Approved`
  - `users.kyc_status = Verified`
  - 同时把 `users.name` 更新为真实姓名
- `rejectKYC(uid)`
  - `kyc_submissions.status = Rejected`
  - `users.kyc_status = Rejected`

## 9.4 充值地址控制

后台通过 `saveDepositAddress()` 写 `deposit_addresses`：

- 以币种为文档 ID
- 下面挂 `networks`
- 每条网络包含：
  - `address`
  - `qrCode`
  - `status`
  - `updatedAt`

前台移动端和桌面端都通过监听 `deposit_addresses` 来展示充值地址。

## 9.5 充值审核

充值流程大致是：

1. 用户在前台看到充值地址
2. 用户上传充值凭证
3. 写入 `deposits`
4. 后台审核

后台通过：

- `handleApproveDeposit(id, uid, amount)`
- `handleRejectDeposit(id)`

进行处理。

批准充值时会：

- 把 `deposits.status = Approved`
- `users.balance += amount`

## 9.6 提现审核

移动端提交提现会写入 `withdrawals`。  
后台通过：

- `handleApproveWithdrawal(id, uid, amount)`
- `handleRejectWithdrawal(id)`

进行处理。

批准提现时会：

- `withdrawals.status = Completed`
- `users.balance -= amount`

## 9.7 客服系统

客服结构为：

- `support_chats/{uid}`：会话主文档
- `support_chats/{uid}/messages`：消息子集合

前台移动端：

- 支持文字消息
- 支持图片消息
- 监听未读数
- 悬浮客服图标可直达聊天

后台：

- 实时监听 `support_chats`
- 选择会话后监听其 `messages`
- 回复消息时写入同一子集合
- 更新 `unread_count_user` / `unread_count_admin`

因此客服是真正的前后台实时联动模块。

## 12. 当前实现的真实特征与风险点

## 12.1 真实特征

这个项目的业务特点非常鲜明：

- 移动端是第一业务入口
- 交易记录、充值提现、客服、KYC 都是 Firestore 实时驱动
- 后台不是独立 API 服务，而是直接改 Firestore 文档
- 交易结算大量在前端完成

## 12.2 需要特别注意的地方

### 不是服务端撮合系统

永续和交割都不是服务器端撮合撮价模型，而是：

- 前端开单
- 前端平仓/到期结算
- Firestore 保存结果
- 后台通过规则控制结果

### 永续结果可被后台追溯修改

后台改 `customClosePrice` 后，甚至可以对已关闭仓位重新结算，并把差额回写余额。

### 交割合约更像“定时涨跌”

它和传统交割期货不一样，更接近带倒计时的方向判断合约。

### 多页面能力集中在 `mobile.html`

`mobile.html` 承担了大量业务，后续维护时应优先把它视为主应用，而不是普通页面。

## 14. 一句话总结

这个项目本质上是一个：

**“以移动端为核心、以 Firebase/Firestore 为实时后端、由前端直接承担交易结算逻辑、并允许后台通过用户规则和交易规则控制永续/交割合约结果”的交易所风格站点。**

如果要继续扩展功能，最重要的三个主轴是：

- `mobile.html` 的业务路由与资金/交易链路
- `admin.html` 的后台控制面板
- `trades` / `users` 两张核心数据表的结算与控制字段
